# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

lighttpd 1.4 is a single-binary, event-driven HTTP/1.x and HTTP/2 web server written in C11. The source lives entirely in `src/` (~100 `.c` files); everything else is build scaffolding, tests, and docs.

## Build systems

Four build systems are kept in sync and all exercised by CI. Pick one; don't mix build artifacts. `scripts/ci-build.sh {meson|cmake|cmake-asan|scons}` is the canonical full-featured invocation for each (it enables every optional dependency) ‚Äî read it to see the exact flags CI uses.

- **autotools** (release builds, `./configure --help` lists all `--with-*` feature flags):
  ```sh
  ./autogen.sh && ./configure -C && make -j4
  ```
- **meson** (fast dev iteration):
  ```sh
  meson setup build && ninja -C build      # or: meson compile -C build
  ```
- **cmake** (`-DWITH_*=ON` for features; `-DBUILD_SANITIZE_ADDRESS=ON -DBUILD_SANITIZE_UNDEFINED=ON` for ASan/UBSan):
  ```sh
  cmake -S . -B build -Wno-dev && cmake --build build -j4
  ```
- **scons** (used for static-link builds): `scons -j4 build_static=1 build_dynamic=0 ...`

`-D build_static=true` (meson) / `LIGHTTPD_STATIC` (autotools, see `INSTALL`) produces a monolithic binary with plugins compiled in instead of loaded as `.so`.

## Running tests

`make check` runs both test suites. There are two distinct kinds:

- **Unit tests** ‚Äî C programs in `src/t/test_*.c`, fast smoke tests, the preferred place to add coverage (see `src/t/README`). Run a single one:
  ```sh
  meson test -C build test_request          # meson
  ctest --test-dir build -R '^test_request' # cmake
  ```
- **Integration tests** ‚Äî Perl harness in `tests/*.t` driving a real lighttpd process. Slower; being gradually replaced by unit tests. Run individually:
  ```sh
  cd tests && ./prepare.sh
  VERBOSE=1 RUNTESTS=request ./run-tests.pl   # or just: ./request.t
  ```
  `tests/LightyTest.pm` is the harness; `TRACE_HTTP=1`, `TRACEME=valgrind|gdb|strace` add diagnostics (see `tests/README`). Integration tests require lighttpd to be built first and several `perl-*` modules (listed in `INSTALL`).

## Architecture

**Process model.** `src/server.c` holds `main()` / `server_main()`: parses config, optionally `fork()`s worker processes, then runs `server_main_loop()`. The loop is driven by `fdevent_poll()` with a 1-second tick that fires per-second `handle_trigger` plugin hooks.

**Event abstraction.** `fdevent.c` / `fdevent_impl.c` wrap the platform poller (epoll / kqueue / poll / select / Solaris ports / Windows). Code registers an fd + callback via `fdevent_register`; everything is non-blocking and callback-driven.

**Connection & request lifecycle.** `connections.c` runs `connection_state_machine()` over the `CON_STATE_*` enum in `src/request.h`:
`CONNECT ‚Üí REQUEST_START ‚Üí READ ‚Üí REQUEST_END ‚Üí READ_POST ‚Üí HANDLE_REQUEST ‚Üí RESPONSE_START ‚Üí WRITE ‚Üí RESPONSE_END` (plus `ERROR`, `CLOSE`). A `connection` is the socket; a `request_st` (`src/request.h`) is one HTTP request. HTTP/1 has one request per connection; HTTP/2 multiplexes many via the `hxcon` struct (`r[8]` arrays of streams). Protocol parsing/framing is split into `h1.c` (HTTP/1.x) and `h2.c` (HTTP/2); `request.c`/`response.c` are version-agnostic.

**Plugin system.** This is the core extension mechanism ‚Äî almost all features are `mod_*.c` plugins. `src/plugin.h` defines the hook vtable; each plugin implements a subset of `handle_uri_raw`, `handle_uri_clean`, `handle_docroot`, `handle_physical`, `handle_subrequest`, `handle_request_done`, `handle_connection_*`, `handle_trigger`, etc., plus lifecycle `init` / `set_defaults` / `worker_init` / `cleanup`. `plugin.c` registers them. Hooks fire at the matching points of the connection state machine; a hook returns a `handler_t` (`HANDLER_GO_ON`, `HANDLER_FINISHED`, `HANDLER_WAIT_FOR_EVENT`, ‚Ä¶) that steers the state machine. `mod_skeleton.c` is the template for a new plugin.

**Configuration.** `configfile.c` loads config; the parser is generated by the bundled **lemon** tool (`lemon.c` builds it; `configparser.y` is the grammar ‚Üí `configparser.c`). Conditional blocks (`$HTTP["host"] =~ ...`) become `data_config` nodes (`data_config.h`); conditions are evaluated per-request to layer a `request_config` (`request.h`) onto the request. PCRE2 (or PCRE) backs regex conditions.

**Gateway backends.** `gw_backend.c` is shared connection/load-balancing/streaming machinery used by `mod_cgi`, `mod_fastcgi`, `mod_scgi`, `mod_ajp13`, `mod_proxy`, `mod_sockproxy`, and `mod_wstunnel`. Touching backend pooling/balancing logic usually means editing `gw_backend.c`, not the individual modules.

**TLS** is pluggable: exactly one of `mod_openssl`, `mod_boringssl`, `mod_mbedtls`, `mod_wolfssl`, `mod_gnutls`, `mod_nss` provides the TLS layer. Keep changes that affect TLS behavior consistent across these backends.

**Core data structures** (used everywhere; understand these before editing hot paths):
- `buffer.c` ‚Äî the central growable string/byte-buffer type.
- `chunk.c` ‚Äî `chunkqueue` of memory *and* file chunks, enabling zero-copy responses (`network_write.c` uses `sendfile` etc.). Request bodies and response output flow through chunkqueues.
- `array.c` ‚Äî ordered associative array; `http_header.c` / `http_kv.c` ‚Äî header storage and well-known string/enum tables.
- `stat_cache.c` ‚Äî cached `stat()` results, optionally backed by FAM/inotify.

## Conventions

- C11, no external C++ runtime. Match the existing terse, allocation-conscious style; reuse `buffer`/`array`/`chunkqueue` rather than raw `malloc`/`char*`.
- Plugins must not mutate `request_st.state` directly (noted in `request.h`); return the appropriate `handler_t` instead.
- A new `mod_*.c` must be wired into all build systems (`src/meson.build`, `src/CMakeLists.txt`, `src/Makefile.am`, `SConstruct`) and, for static builds, the `plugin-static.h` mechanism described in `INSTALL`.
- `NEWS` is the human-facing changelog; user-visible changes get an entry.

## BSC Project Compile Command

BiSheng C pilot translation lives in `src/buffer.c` / `src/buffer.h`. Annotations are
macro-guarded by `src/bsc_compat.h` so plain-C builds are unaffected.

- BSC compiler: `~/bsc/llvm-project/install/bin/clang` (clang 15.0.4, supports `-x bsc`)
- libcbs include: `/home/zly/bsc/llvm-project/install/include/libcbs`
- runtime lib (link only): `/home/zly/bsc/llvm-project/install/lib` (`-lstdcbs`)

Generate config.h once (meson/autotools absent here; cmake present):
    cmake -S . -B build-cmake -Wno-dev -DWITH_PCRE2=OFF -DWITH_PCRE=OFF

Verify (borrow-check buffer.c, no link):
    ~/bsc/llvm-project/install/bin/clang -x bsc -DHAVE_CONFIG_H \
      -Ibuild-cmake/build -Isrc \
      -I/home/zly/bsc/llvm-project/install/include/libcbs \
      -Wno-nullability-completeness -fsyntax-only src/buffer.c

Includers unaffected (buffer.h stays dual-valid): all standalone `src/*.c`
except buffer.c parse clean under gcc; failures are only missing optional-dep
headers / Windows / lemon templates, never buffer.h. Spot-check:
    gcc -DHAVE_CONFIG_H -Ibuild-cmake/build -Isrc -fsyntax-only src/base64.c

Test (behavioral parity): a translated buffer.c is a BSC-only TU, so `ctest`
(which compiles buffer.c with gcc) cannot run it. Build the runner directly ‚Äî
buffer.o under -x bsc, gcc test harness using buffer.h, linked with -lstdcbs:
    BSC=~/bsc/llvm-project/install/bin/clang
    LIBCBS=/home/zly/bsc/llvm-project/install/include/libcbs
    LIBDIR=/home/zly/bsc/llvm-project/install/lib; W=/tmp/bsc_test_buffer; mkdir -p $W
    $BSC -x bsc -DHAVE_CONFIG_H -Ibuild-cmake/build -Isrc -I$LIBCBS \
      -Wno-nullability-completeness -c src/buffer.c -o $W/buffer.o
    sed 's|#include "buffer.c"|#include "buffer.h"|' src/t/test_buffer.c > $W/tb.c
    printf 'void test_buffer(void);\nint main(void){test_buffer();return 0;}\n' > $W/m.c
    gcc -DHAVE_CONFIG_H -Ibuild-cmake/build -Isrc -c $W/tb.c -o $W/tb.o
    gcc -DHAVE_CONFIG_H -Ibuild-cmake/build -Isrc -c src/ck.c -o $W/ck.o
    gcc -DHAVE_CONFIG_H -Ibuild-cmake/build -Isrc -c $W/m.c  -o $W/m.o
    $BSC $W/tb.o $W/buffer.o $W/ck.o $W/m.o -L$LIBDIR -lstdcbs -o $W/run && $W/run
(The repo's src/t/test_buffer.c is NOT modified; the include swap is only in the
throwaway harness copy, which links fine because _Owned/_Borrow erase at codegen.)
